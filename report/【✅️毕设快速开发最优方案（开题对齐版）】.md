# JeecgBoot校园问答系统毕设快速开发最优方案（开题对齐版）

## 1. 方案定位

### 1.1 目标

在不超纲、不拖期的前提下，完成一个“可演示、可验收、可答辩”的校园问答系统。

- 可演示：学生端和后台形成完整闭环
- 可验收：覆盖开题报告核心功能点
- 可答辩：有明确技术亮点、有数据和测试材料支撑

### 1.2 核心策略

坚持“**完整业务闭环 + 轻量智能能力 + 材料完整**”三条线，避免进入企业级复杂度陷阱。

- 主线：JeecgBoot 单体架构 + Vue3 后台 + 微信小程序
- 智能能力：关键词检索 + 简易意图识别（规则/Jieba）
- 不把 DeepSeek/Milvus 作为必交内容

## 2. 开题要求与当前状态对齐

| 开题要求 | 当前状态判断 | 交付目标 |
| --- | --- | --- |
| 小程序学生端 | 已有首页/搜索/详情/通知/指南基础页，登录和个人中心仍占位 | 补齐登录绑定、个人中心、收藏/历史/反馈闭环 |
| Web管理后台 | `knowledge/notice/guide/category/feedback` 已搭建，中文化已完成 | 稳定 CRUD、补统计展示 |
| 后端服务 | `jeecg-boot-module-campusqa` 模块已成型并可编译 | 补齐 mini 侧缺口接口、统一校验和返回 |
| 智能问答 | 现为关键词检索 + 简单 `rag` 包装 | 保持“轻量智能”定位，加入规则意图识别即可达标 |
| 权限与角色 | Jeecg 自带 RBAC 可用，菜单问题已修复 | 明确学生/信息员/管理员操作边界并留痕 |
| 数据与统计 | 核心表和业务链存在，统计薄弱 | 补 1 个统计页（访问、热门、反馈处理率） |

## 3. 毕设最优范围（建议最终版）

## 3.1 Must（必须完成）

1. 学生端：
   - 问答搜索、问答详情
   - 通知列表、办事指南列表
   - 收藏、历史、反馈提交
2. 信息员端：
   - 问答库管理、分类管理、通知管理、指南管理
   - 反馈查看与回复
3. 管理端：
   - 基于 Jeecg 完成用户/角色/菜单权限管理
4. 智能能力：
   - 关键词检索
   - 简易意图识别（规则 + 关键词词典）
5. 统计能力：
   - 热门问题 TOPN
   - 反馈处理率
   - 通知阅读/浏览计数（可简化为点击计数）

## 3.2 Should（建议完成）

1. 敏感词拦截（问答/通知提交时）
2. 统一异常处理和提示文案
3. 关键接口参数校验

## 3.3 Could（有余力再做）

1. 微信订阅消息真实推送
2. Excel 批量导入问答
3. 更细粒度可视化报表

## 3.4 Won’t（本次不做）

1. 生产级向量检索链路（Milvus + rerank + 多模型调度）
2. 微服务拆分与云原生部署
3. 大规模压测与高可用治理

## 4. 快速开发任务拆分（按代码目录）

## 4.1 后端（JeecgBoot）

目录：`jeecg-boot/jeecg-boot-module/jeecg-boot-module-campusqa`

1. `mini` 控制器补全学生端闭环接口  
   参考文件：`.../mini/controller/CampusQaMiniController.java`
   - 新增：历史列表、收藏列表、订阅列表
   - 细化：反馈提交参数校验、收藏/订阅去重逻辑
2. 智能问答轻量增强  
   参考文件：`.../rag/service/impl/RagServiceImpl.java`
   - 增加规则意图识别（教务/学工/后勤/办事）
   - 输出更合理置信度和命中说明
3. 统计接口  
   - 新增一个 `stats` 控制器或挂到现有控制器
   - 提供 TOP 热门问题、反馈处理率、问答总量
4. 审计与安全  
   - 保留 Jeecg 操作日志
   - 对写接口加基本参数校验和错误提示

## 4.2 管理后台（Vue3）

目录：`jeecgboot-vue3/src/views/campusqa`

1. 已完成项：菜单显示、页面中文化、基础 CRUD
2. 需补项：
   - 增加“统计概览”页面（可单页卡片+表格）
   - 操作反馈文案统一（成功/失败/删除确认）
3. 权限验证：
   - 管理员可见全量菜单
   - 信息员角色仅见本业务菜单

## 4.3 小程序（学生端）

目录：`campus-qa-miniapp`

1. 必改页面（当前仍有占位痕迹）：
   - `pages/login/index.js`、`pages/login/index.wxml`
   - `pages/profile/index.js`、`pages/profile/index.wxml`
2. 功能闭环：
   - 登录后持久化 `userId`（先模拟绑定，不阻塞毕业设计）
   - 个人中心展示：历史、收藏、反馈入口
   - 问答详情页支持：收藏、反馈按钮
3. 体验统一：
   - `app.json` 页面标题和 tab 文案改中文
   - 错误提示、空态提示统一

## 4.4 数据与菜单

目录：
- `jeecg-boot/db/campusqa-menu.sql`
- `jeecg-boot/db/jeecgboot-mysql-5.7.sql`

要求：
1. 菜单 SQL 保持幂等
2. `sys_permission.parent_id` 顶级菜单为空字符串
3. `admin/vue3` 角色授权完整

## 6. 验收标准（毕设可通过版）

## 6.1 功能验收（必须全部通过）

1. 学生可搜索并查看问答详情
2. 学生可查看通知与办事指南
3. 学生可收藏问答并查看历史
4. 学生可提交反馈
5. 信息员可维护问答、通知、指南、分类
6. 信息员可回复反馈
7. 管理员可配置角色与菜单权限
8. 后台页面文案中文化完整
9. 菜单权限在不同角色下显示正确
10. 统计页可展示热门问题与反馈处理率

## 6.2 性能与稳定（简化指标）

1. 常用查询接口响应时间：本地环境平均 < 1.5s
2. 连续演示 20 次操作无报错中断
3. 服务重启后菜单、数据、权限保持一致

## 6.3 材料验收

1. 系统架构图 + 功能结构图
2. 数据库 ER 图 + 核心表说明
3. 关键功能截图（学生端/后台）
4. 接口清单与测试记录
5. 演示脚本（按场景）

## 7. 答辩演示主线（建议直接照此讲）

1. 背景痛点：校园信息分散、查询低效
2. 方案：微信小程序入口 + JeecgBoot 管理后台 + 轻量智能检索
3. 演示链路：
   - 学生提问 -> 命中答案
   - 后台修改知识 -> 学生再次查询生效
   - 学生反馈 -> 信息员回复 -> 闭环完成
4. 结果指标：命中率、热门问题、反馈处理率
5. 创新点：低成本实现“智能问答 + 业务管理 + 数据闭环”

## 8. 风险与应对

1. 风险：小程序登录链路耗时长  
   应对：先做“本地 userId 绑定”保证闭环，真实授权列为可选增强
2. 风险：智能问答效果不稳定  
   应对：保底关键词检索 + 规则意图识别，避免空答
3. 风险：功能太多导致延期  
   应对：严格执行 Must/Should/Could，不做超纲项
4. 风险：答辩时演示故障  
   应对：准备演示数据库快照与固定测试账号
