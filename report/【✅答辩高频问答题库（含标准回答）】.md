# 答辩高频问答题库（含标准回答）
版本：v1.0  
日期：2026-02-21  
项目：JeecgBoot校园问答系统

---

## 一、选题与目标类

### 1. 你为什么选这个题目？
校园咨询信息分散、重复咨询多、响应慢。我做这个系统是为了解决“查询入口分散 + 人工答疑压力大”的问题，做一个可落地的校园问答闭环系统。

### 2. 你的系统核心目标是什么？
三个目标：学生端可查可问可反馈；管理端可维护可运营；系统有轻量智能和统计能力，能支撑实际演示与验收。

### 3. 你的创新点是什么？
不是算法堆料，而是“轻量意图识别 + 闭环数据回流 + 可运营统计看板”，在毕设范围内实现了完整工程闭环。

---

## 二、架构与技术选型类

### 4. 为什么用 JeecgBoot？
JeecgBoot 自带 RBAC、字典、日志、代码生成和后台管理能力，能显著降低开发成本，适合毕业设计在有限周期内交付完整系统。

### 5. 为什么不用微服务？
毕业设计优先可交付和稳定性。单体架构更容易开发、联调和答辩演示，复杂度可控。

### 6. 为什么前端是 Vue3 + 小程序双端？
后台管理和学生使用场景不同：管理端适合 Web，学生端适合微信小程序。双端能完整覆盖业务闭环。

### 7. 数据库为什么选 MySQL + Redis？
MySQL 负责结构化持久化，Redis 负责缓存与会话。这个组合成熟稳定，运维门槛低。

---

## 三、功能实现类

### 8. 系统有哪些核心模块？
知识库、分类、通知、办事指南、反馈、历史、收藏、订阅、轻量问答、统计概览。

### 9. 学生端闭环具体怎么体现？
搜索问答 -> 查看详情 -> 收藏/反馈 -> 记录历史 -> 个人中心查看收藏历史订阅。

### 10. 管理端闭环具体怎么体现？
管理员维护知识和通知，处理反馈；学生侧行为写入历史；统计页回看热门问题和意图分布。

### 11. 你怎么处理权限控制？
基于 JeecgBoot 的角色-菜单-权限模型，菜单和按钮权限都通过 `sys_permission` 与 `sys_role_permission` 控制。

### 12. 你怎么保证“校园问答菜单”稳定显示？
使用幂等 SQL 初始化菜单与权限，修复 parent_id、menu_type、角色授权，联调后用管理员和 vue3 角色双重验证。

---

## 四、智能问答与算法类

### 13. 你的智能问答是大模型直出吗？
不是。当前是“轻量意图识别 + 规则/检索优先”的方案，强调稳定、可解释、可落地。

### 14. 意图识别怎么做的？
基于校园场景关键词规则划分意图类型，如教务、学工、后勤、流程、时间、地点、综合，并输出意图分值。

### 15. 为什么不用重型向量检索方案？
重型方案（向量数据库 + 重排序 + 多模型）复杂度和风险较高，不适合作为本次毕设主交付。当前方案已满足开题要求并可稳定演示。

### 16. 你如何支撑后续扩展到更强智能？
现有 `rag` 结构已预留扩展点，后续可接入向量召回、重排序和外部大模型，不影响现有业务接口。

## 五、测试与验收类

### 17. 你怎么证明系统是可用的？
有里程碑进度表、联调验收记录、接口返回证据、后端日志、统计数据回流结果，且支持脚本化复现。

### 18. 关键通过的接口有哪些？
`/campusqa/mini/ask`、`/campusqa/stats/overview`、`/campusqa/stats/topQuestions`、`/campusqa/stats/intentStats`、`/campusqa/mini/subscribe/list`。

### 19. 你做了哪些稳定性处理？
接口参数校验、防空与分页边界处理、小程序请求层统一错误拦截、联调脚本自动化。

### 20. 如何避免脏数据？
在 mini 写接口层进行字段必填和长度约束，如反馈内容必填、历史写入至少包含 query 或 knowledgeId。

## 六、难点与问题处理类

### 21. 你项目里最难的问题是什么？
菜单权限和联调一致性问题。根因是菜单 SQL、角色授权和运行时依赖版本不一致。通过 SQL 幂等修复 + 重新安装模块依赖解决。

### 22. `vue-tsc` 报错你怎么处理的？
先处理阻塞性语法错误（`MyComment.vue` 的 TS1003），保证本次功能可联调；历史类型问题作为遗留清单单独处理。

### 23. 你为什么做里程碑推进？
能把“开发-联调-验收-文档”闭环化，每阶段可验证、可回溯，降低临近答辩时的风险。

---

## 七、论文与工程价值类

### 24. 你的工程价值在哪里？
在有限时间内实现了可运行、可维护、可统计的校园问答系统，不是原型图，而是可联调的完整工程。

### 25. 你的论文价值在哪里？
给出了从需求到实现再到验收的完整路径，尤其是轻量智能策略在校园场景下的工程化落地方法。

### 26. 你的不足是什么？
当前智能策略以规则为主，语义泛化能力有限；小程序登录采用本地 userId 方案，未接入完整微信身份体系。

### 27. 后续怎么改进？
接入真实微信登录链路、增强语义检索与重排序、丰富统计图表并加入更细粒度运营指标。

## 八、收口万能回答（临场可用）

### 28. 如果老师问“你这个算不算 AI”？
算轻量 AI 工程应用。我的目标是“在毕设周期内可落地、可验证”，所以选择可解释、可复现的意图识别与检索增强策略。

### 29. 如果老师问“和现有 FAQ 有什么区别”？
区别在于闭环：不只是静态问答展示，而是有收藏、反馈、历史、订阅和后台统计，能持续优化内容。

### 30. 如果老师问“你个人贡献是什么”？
我完成了校园问答模块的后端实现、前端页面联动、菜单权限修复、里程碑联调脚本和验收文档沉淀，形成完整可答辩交付。

## 九、使用建议
1. 先背“前 12 题”，覆盖 80% 提问。  
2. 技术细节优先讲“你做过并验证过”的点。  
3. 回答结构固定：问题背景 -> 你的决策 -> 验证结果。  
4. 遇到不会的问题，用“当前实现 + 后续优化”方式稳住节奏。
