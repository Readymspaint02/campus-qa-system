## 一、答辩演示脚本（可直接照读，约 10-12 分钟）

### 1. 开场（0:00 - 0:40）
各位老师好，我的毕业设计题目是《基于 JeecgBoot 的校园问答系统设计与实现》。  
本系统面向校园高频咨询场景，目标是把“学生端查询 + 管理端维护 + 轻量智能问答 + 数据统计”形成完整闭环。

### 2. 研究目标与完成情况（0:40 - 1:30）
我的开题目标主要有三点：
1. 学生端能快速查问答、看通知、提反馈。  
2. 管理端能维护知识库、分类、通知、办事指南，并处理反馈。  
3. 系统具备轻量智能能力和统计分析能力。  

目前三项均已完成，并完成了分里程碑联调和验收记录。

### 3. 系统架构说明（1:30 - 2:40）
系统采用三层结构：
1. 展示层：微信小程序 + Vue3 后台。  
2. 业务层：JeecgBoot 单体应用，校园问答独立模块 `jeecg-boot-module-campusqa`。  
3. 数据层：MySQL（业务数据）+ Redis（缓存与会话）。  

智能问答采用“轻量规则意图识别 + 知识库检索”的方案，兼顾实现难度和毕设可落地性。

### 4. 核心功能演示脚本（2:40 - 7:40）
下面按“学生使用 -> 后台维护 -> 统计回看”的闭环演示。

#### 4.1 学生端（小程序）
1. 打开首页，输入关键词搜索问题。  
2. 进入问答详情页，展示答案内容。  
3. 点击收藏，提交反馈。  
4. 进入个人中心，查看“我的收藏、查询历史、我的订阅”。  

说明：登录采用本地 `userId` 绑定方式，满足毕设闭环演示需求。

#### 4.2 管理端（Vue3）
1. 进入“校园问答”菜单。  
2. 展示知识库、通知公告、办事指南、分类管理、反馈管理页面。  
3. 在反馈管理页面查看用户反馈并处理。  

#### 4.3 统计分析（里程碑2/3成果）
1. 打开“统计概览”页面。  
2. 展示指标：知识库总量、反馈处理率、历史问答量等。  
3. 展示热门问题 TopN。  
4. 展示意图分布统计（基于 `ask:*` 历史来源聚合）。

### 5. 关键技术亮点（7:40 - 9:00）
1. 轻量意图识别：按校园场景定义意图类型（教务、学工、后勤、流程、时间、地点、综合）。  
2. 闭环数据链路：问答请求会回写历史，统计端可直接聚合展示。  
3. 工程化能力：里程碑化推进，每个里程碑有“进度表 + 联调验收记录 + 可复现脚本”。

### 6. 测试与结果（9:00 - 10:00）

3. 里程碑3新增稳定化校验通过：空反馈拦截、无效历史请求拦截、订阅列表联通。

### 7. 结尾（10:00 - 10:30）
本系统完成了开题要求的核心功能，具备可演示、可验收、可答辩的完整度。

## 三、论文“系统实现”章节可粘贴稿（可直接放第4章/第5章）

以下内容可直接粘贴进论文，再按学校模板调整图号、表号和参考文献编号。

### 4.1 系统实现总体说明
本系统采用 JeecgBoot + Vue3 + 微信小程序的前后端分离架构。后端在 JeecgBoot 基础上扩展校园问答业务模块，前端由管理端和学生端组成。管理端负责知识维护与数据治理，学生端提供问答查询、收藏反馈与个人中心能力。系统通过统一接口协议实现业务闭环，保证功能可演示、可验收、可持续扩展。

从工程组织上，后端核心代码位于 `jeecg-boot-module-campusqa`，管理端页面位于 `jeecgboot-vue3/src/views/campusqa`，小程序代码位于 `campus-qa-miniapp/pages`。项目采用里程碑方式推进，分别完成基础业务闭环、统计能力建设和稳定化收尾。

### 4.2 后端核心模块实现
#### 4.2.1 业务域模型与接口设计
系统围绕知识库、分类、通知、办事指南、反馈、历史、收藏、订阅等业务实体展开设计。面向学生端提供 mini 系列接口，面向管理端提供后台 CRUD 与统计接口。  
mini 接口覆盖：
1. 知识检索与详情查询。  
2. 历史记录写入与查询。  
3. 收藏切换与收藏列表。  
4. 反馈提交。  
5. 订阅切换与订阅列表。  
6. 问答入口 `/mini/ask`。  

统计接口覆盖：
1. `overview`：总量与处理率指标。  
2. `topQuestions`：热门问题 TopN。  
3. `intentStats`：意图类型分布统计。

#### 4.2.2 轻量智能问答实现
系统智能问答采用轻量方案：规则意图识别 + 知识库检索。意图类型包括教务咨询、学工咨询、后勤咨询、流程、时间、地点与综合咨询。问答结果返回时包含意图标签、置信分数等辅助字段，便于前端展示和后续统计分析。  

在问答接口执行后，系统会将请求写入历史表，并将来源标记为 `ask:INTENT_TYPE`。该设计使统计模块能够直接基于历史记录聚合意图分布，实现低耦合的数据闭环。

#### 4.2.3 稳定化与参数校验
为提升线上可用性，系统在 mini 关键接口增加参数校验和边界保护：
1. 分页参数统一归一化处理。  
2. 详情接口要求必填主键参数。  
3. 反馈内容不能为空且长度受限。  
4. 历史写入要求至少提供查询词或知识ID。  
5. 订阅切换要求至少提供院系或分类条件。  

该类校验可有效降低无效请求导致的脏数据风险，也提升了接口行为可预测性。

### 4.3 前端实现
#### 4.3.1 管理端（Vue3）实现
管理端基于 Vue3 与 Jeecg 组件体系实现校园问答业务页面，包括知识库管理、通知公告、办事指南、分类管理、反馈管理与统计概览。  
统计概览页对接后端统计接口，展示关键指标卡片、热门问题列表和意图分布数据，为管理者提供运营观察能力。

#### 4.3.2 学生端（小程序）实现
学生端页面包含首页、搜索、详情、通知、指南、登录与个人中心。系统实现“查询-详情-收藏-反馈-历史-个人中心”闭环。  
个人中心页面可展示查询历史、收藏列表和订阅列表。登录采用本地 `userId` 绑定方案，以满足毕设阶段的快速开发与稳定演示需求。

#### 4.3.3 请求层统一处理
在小程序侧封装统一请求工具，对 HTTP 状态码异常和业务失败返回进行统一拦截，避免页面重复编写错误处理逻辑。该实现提升了前端代码一致性和可维护性。

### 4.4 系统测试与联调结果
系统采用“模块编译 + 接口联调 + 里程碑验收”方式完成测试：
1. 后端模块编译通过。  
2. 关键接口联调通过（问答、统计、订阅、校验场景）。  
3. 形成里程碑进度与验收文档，测试过程可追溯。  

联调结果表明系统已具备稳定演示能力：问答接口可返回结果并写入历史，统计接口可实时反映累计数据，参数校验可正确拦截无效输入。

### 4.5 本章小结
本章完成了系统从后端模块、前端页面到联调测试的实现说明。实践结果表明，该系统在毕设范围内实现了“业务闭环 + 轻量智能 + 数据统计”的目标，能够支撑后续答辩演示与论文验收。
